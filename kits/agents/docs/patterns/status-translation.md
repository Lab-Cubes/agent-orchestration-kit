# Status Translation Pattern

Dispatch has two result sources:

- A worker-written NOP `result.json`.
- A synthetic fallback result generated by `cmd_dispatch` when the worker did
  not write one.

The distinction matters because only worker-written results can express
domain-level outcomes like `blocked` with a `pushback_reason`.

## Runtime Stream To CSV Status

`cmd_dispatch` first parses the runtime stream output. It extracts the clean
JSON result event and derives `status_val` for the operator-facing CSV:

- `status_val=success` when the runtime result has `is_error=false`.
- `status_val=error` when `is_error=true`, parsing fails, or no JSON is found.

See `kits/agents/scripts/lib/cmd_dispatch.sh:414-462`.

`status_val` is not the NOP wire status. It is the dispatch log status used for
cost rows and hooks.

## Synthetic Fallback Result

If the worker does not write `done/<task>.result.json`, `cmd_dispatch` generates
a fallback result (`kits/agents/scripts/lib/cmd_dispatch.sh:474-512`).

The fallback maps `status_val` and stop reason to NOP payload status at
`kits/agents/scripts/lib/cmd_dispatch.sh:494`:

| Condition | Fallback `payload.status` |
|---|---|
| `status_val == "success"` | `completed` |
| `stop_reason == "time_limit"` | `timeout` |
| Anything else | `failed` |

Fallback payloads also include:

- Empty `files_changed` and `commits`.
- A `follow_up` entry telling the operator to review raw output.
- `_fallback=true`.
- `_turns`.
- `_stop_reason`.

Synthetic fallback cannot produce `blocked`. It has only runtime stream status
and stop reason, not a worker decision or `pushback_reason`.

## Worker-Written Result

When the worker writes `done/<task>.result.json`, that file is the NOP result
artifact. This path is required for outcomes that are meaningful inside the NOP
protocol, especially:

- `status=blocked` with `pushback_reason`.
- `files_changed` for scope validation.
- `commits` and `follow_up` produced by the worker.

The mock worker uses this pattern for `scope_violation`, `scope_clean`,
malformed-result cases, and `pushback`
(`kits/agents/tests/bin/claude:114-172`,
`kits/agents/tests/bin/claude:215-280`).

## Plan Stamping And Result Validation

Task-list dispatch owns the plan boundary. If `--plan-id` was supplied,
`cmd_dispatch` stamps `result.payload.plan_id` after the worker or fallback
result exists (`kits/agents/scripts/lib/cmd_dispatch.sh:515-524`).

The dispatcher then checks for required NOP fields. If `result.json` is
malformed or missing `_ncp`, `payload`, or `payload._nop`, it marks
`status_val=error` (`kits/agents/scripts/lib/cmd_dispatch.sh:527-535`).

## Scope Validation Rewrite

Scope validation compares `result.payload.files_changed` against the original
`constraints.scope` (`kits/agents/scripts/lib/cmd_dispatch.sh:537-575`).

On a violation, dispatch:

- Logs the violating paths.
- Sets CSV `status_val=error`.
- Sets `result.payload.status=failed`.
- Appends a scope-violation message to `result.payload.error`.
- Adds `result.payload._scope_violation=true`.

The comments at `kits/agents/scripts/lib/cmd_dispatch.sh:560-563` capture the
reason for the split: CSV `status_val` is operator-facing, while
`result.payload.status` must remain a valid NOP task status.

## Task-List Node Outcome Mapping

`cmd_dispatch_tasklist` reads the worker result after direct dispatch returns.
It extracts the task id from dispatch logs, finds the result file, and reads
`result.payload.status` plus `pushback_reason`
(`kits/agents/scripts/lib/cmd_dispatch_tasklist.sh:651-682`).

The mapping is intentionally small:

| Result payload | Node outcome |
|---|---|
| `status == "completed"` | `completed` |
| `status == "blocked"` and `pushback_reason` is present | `pushback` |
| Anything else | `failed` |

When the node outcome is `pushback`, task-list dispatch marks the node blocked,
logs the pushback, and invokes the decomposer with prior version, prior state,
and the worker reason
(`kits/agents/scripts/lib/cmd_dispatch_tasklist.sh:684-713`).

## Practical Rule

Use runtime stream assertions for adapter behavior, token accounting, timeout,
or fallback tests. Use worker-written `result.json` fixtures when the behavior
depends on NOP payload semantics such as pushback, scope validation, malformed
wire artifacts, commits, or follow-up tasks.
